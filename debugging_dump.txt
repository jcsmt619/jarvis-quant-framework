Guide

What this does: Gives Claude Code a reference doc to consult when your bot breaks. Claude loads it automatically when you're stuck. Structured diagnostics instead of generic guesses.

What's Inside

Covers the 8 most common bot failure modes:

1. Backtest Sharpe unrealistically high (>2.5)
   Walks through look-ahead bias, slippage, training contamination.

2. Paper trading performance differs from backtest
   Diagnoses whether paper is worse (slippage, look-ahead) or better
   (unrealistic Alpaca fills, wrong strategy).

3. HMM flickering between regimes
   Too many regime candidates, insufficient training data, noisy features.

4. Circuit breaker fired and you don't know why
   Which breaker, how to read the lock file, when to reset.

5. Orders being rejected by broker
   Table of every rejection reason and its fix.

6. Bot crashed overnight
   Log locations, state snapshot, reconciliation, process manager recovery.

7. Dashboard shows UNCERTAIN permanently
   HMM not trained, confidence below threshold, flicker lock.

8. Look-ahead test failing on code you think is correct
   Warmup NaN handling, AST pattern hits, comment false-positives.
How to Install

Drop debug-playbook.md into a docs/ folder in your bot project.

If your CLAUDE.md has this line at the bottom:

@docs/debug-playbook.md
Claude Code loads the playbook automatically whenever it's relevant.

If you don't have that line yet, tell Claude Code:

Add @docs/debug-playbook.md to the imports section of my CLAUDE.md 
so you load it automatically when I hit issues.
How to Use It

Describe the symptom to Claude Code in plain English:

"My backtest Sharpe is 3.8 over 3 years, that seems too high"
→ Claude walks the Sharpe-too-high decision tree

"The bot crashed at 3am, how do I figure out what happened"
→ Claude walks the crash diagnostic steps

"Orders keep getting rejected with insufficient_buying_power"
→ Claude pulls the exact fix from the rejection table
Prerequisites

- HMM bot architecture from the main prompt
- CLAUDE.md installed (for automatic loading)
- Claude Code installed Debug Playbook

Common problems and what actually fixes them. Find your symptom, try the fixes in order.

Backtest Sharpe is above 2.5
Almost always look-ahead bias.

Run pytest tests/test_look_ahead.py -v. If it fails, that's it.

If it passes, check these in order:

Grep for hmm.predict or model.predict in core/ and data/. The only approved predict is predict_regime_filtered. Viterbi looks at the whole sequence.

Check how features are standardized. If you see StandardScaler().fit_transform(full_df) anywhere, that uses future data to compute the mean. Use rolling z-scores with a 252-period window.

Grep for .shift(- in core/ and data/. Any negative shift pulls future values to the present.

Check your slippage setting. Default 5bps is optimistic for anything smaller than SPY. Try 10-15bps and rerun.

Is the backtest end date within 6 months of HMM training? That's training contamination.

If all that checks out and Sharpe is still 2.5+, reproduce the backtest in a clean environment with a different seed before believing it.

Paper trading is worse than backtest
Slippage model was too optimistic. Check fill prices in the trade log against bar closes. If fills are systematically worse, raise slippage in settings.yaml.

Backtest might have immediate fills, paper has 1-bar delay. Check your backtester config.

Run the look-ahead test again. This is where hidden bias shows up.

Survivorship bias in your backtest universe. The tickers you tested probably don't include delisted ones.

Paper trading is better than backtest
Alpaca paper fills at mid-price. Real fills are at ask/bid. Live will be worse than paper.

You're running a different strategy than you backtested. Diff the code.

Different data source between backtest and paper. Check the data feed.

HMM flickering between regimes
Regime changes every few bars, flicker rate above threshold.

Too many regime candidates. BIC picked something overfit. Set n_candidates: [3, 4] in settings and retrain.

Not enough training data. HMM needs 2+ years of daily bars (504) to stabilize.

Features too noisy. Any feature with a lookback under 5 periods is noise. Remove or smooth.

Real regime change happened and the model is lagging. Retrain on recent data.

Circuit breaker fired
Check logs/alerts.log. Every fire logs the breaker type, actual DD, equity, and HMM regime at the time.

If the daily DD breaker fired but you think you didn't lose that much, the DD is measured from daily peak equity, not from open. Intraday highs matter.

If peak DD breaker wrote a lock file, read the file contents first. Understand what happened. Then delete the file manually. Do not skip the review step. That's the whole point of the lock.

Orders getting rejected
insufficient_buying_power  -> leverage over broker limit, reduce max_leverage
not_tradeable              -> symbol halted or delisted, remove from list
wash_trade                 -> opposite direction too soon, increase duplicate window
market_closed              -> is_market_open() returning wrong value, check logic
invalid_stop               -> stop on wrong side of price or too close
bid_ask_spread_too_wide    -> your own pre-trade check, wait or skip
Bot crashed overnight
Last entry in logs/main.log tells you what it was doing.

Check logs/alerts.log for what fired right before.

state_snapshot.json timestamp shows if it exited clean (saved state) or was killed.

On Linux, journalctl -u your-bot-service shows OOM kills and systemd restarts.

Check broker web UI for orphan positions the bot didn't reconcile on startup.

Prevention: systemd or supervisord auto-restart, reconcile positions on startup before doing anything else.

Dashboard stuck on UNCERTAIN
HMM not trained. Run python main.py --train-only.

Confidence below threshold. Check logs/regime.log for actual probability values. If they're always low, retrain with more data or lower min_confidence.

Flicker rate exceeded. System locked in uncertainty mode. Check get_regime_flicker_rate().

Bot frozen, can't tell if it's running
Dashboard refreshing = main loop alive.

Tail logs/main.log. New entry every bar means it's running.

Dashboard frozen but logs moving = dashboard thread is dead. Restart.

Logs not moving = main loop stuck. Usually a hung WebSocket reconnect. Send SIGTERM, wait for clean shutdown, restart.

Look-ahead test fails on code that looks correct
Read the test error. It names the value that differed.

Specific feature flagged: that feature uses future data. Look for fit_transform, negative shifts, centered windows, bfill.

Fails on some bars, passes on others: warmup issue. The first N bars of rolling computations are NaN. If you fill those with forward-looking values the test catches it. Accept the NaN.

Fails on a pattern inside a comment or docstring: the AST scan is too aggressive, file an issue or tighten the pattern.

Getting help
Include in your post:

- Exact error message and full traceback
- Which skill or command triggered it
- Last 50 lines of the relevant log
- What you already tried
- Python version and pip freeze output
